WebApp Rendering

October 10, 2020

Il y a peu, en regardant une conférence du CEO de Netlify, j’ai eu un mind blow à propos du rendering que nous mettons en place dans les applications web (API ou Graphiques).

Je me suis rendu compte qu’à données identiques, le code de rendering exécute sans relâche les même opérations pour un contenu de sortie restant toujours identique également.

Un petit exemple valant souvent mieux qu’un long discours, imaginons le point API suivant :

def get_post(post_id: int):
    post: Post = post_repository.load(post_id)

    comments = []
    for comment_id in post.comment_ids:
        comments.append(comment_repository.load(comment_id))


    return {
        "post_id": post.post_id,
        "title": post.title,
        "content": post.content,
        "comments": [
            {
                "author": comment.author,
                "text": comment.text,
                "timestamp": comment.timestamp
            }
            for comment in comments
        ]
    }

Ainsi, à moins qu’une modification ne soit faite sur le Post, ce code retournera toujours la même structure. Hors, si pendant un temps les données d’un Post sont fréquemment modifiées, elles ne le sont pas éternellement. Pire, les données sont probablement bien plus souvent lues que modifiées.

Il semblerait donc que tout code ressemblant de prêt ou de loin à celui-ci augmente la pression sur la base de données utilisée (un nombre conséquent de requêtes pouvant être effectuées) et ne fait que consommer sans cesse des cycles CPU pour un contenu qui est par essence statique. Ces cycles, sans y paraitre, obligent l’entreprise à utiliser des serveurs de capacité plus importante ou encore à en utiliser plus. Outre l’évident impact économique, de ce gaspillage énergétique, un impact écologique est également relevé.

Je reconnais être le premier à tomber dans ce piège (peut être même suis-je le seul ?) mais pour éviter tout lynchage en place publique, intéressons nous plutôt aux moyens de corriger cette erreur de design.

La première solution que j’aborderai est simple. À chaque modification de notre Post, on en fait le rendering et on sauve ce format en base de données pour une utilisation future. Voici ce que cela donnerait :

def on_post_modified(post: Post):
    comments = []
    for comment_id in post.comment_ids:
        comments.append(comment_repository.load(comment_id))


    rendered_post = {
        "post_id": post.post_id,
        "title": post.title,
        "content": post.content,
        "comments": [
            {
                "author": comment.author,
                "text": comment.text,
                "timestamp": comment.timestamp
            }
            for comment in comments
        ]
    }

    rendered_post_table.save(post.post_id, rendered_post)

On suppose ici que ce code est appelé de manière événementielle à chaque modification d’un Post. Le reste du code est assez explicite pour que je me permette de ne pas le commenter.

Avec une telle mécanique mise en place, la point d’API prend alors la forme suivante :

def get_post(post_id: int):
    return rendered_post_table.load(post_id)

Le voilà bien simplifié n’est-ce pas ?

Cette solution a les avantages suivants :

  • Aucun code de rendering n’est exécuté lors de la demande d’accès aux données, elles sont fournies telle quelle;
  • Le nombre de requêtes à la base de données est réduit à 1;
  • On diminue la charge serveur et la pression sur la base de données;
  • On rend plus rapide le retour des données pour l’utilisateur.

Tout ceci semble donc optimal. Cependant, cette solution ne fait selon moi que la moitié du chemin. En effet, si le contenu est statique, pourquoi le stocker dans une base de données ? Cette manière de faire demande deux connexions réseaux. La première entre le client et l’application et la seconde entre l’application et la base de données. Ce faisant, nous ne diminuons pas notre charge serveur autant que nous le pourrions.

La seconde solution que je propose est donc de générer ces données statiques sous forme de fichier et d’utiliser votre application pour les servir sous cette forme.

Pour ce qui est de la génération des fichiers, bonne nouvelle, le code de la méthode événementielle ne doit pas être modifié. Il vous suffit d’écrire le fichier sur disque (ou sur Amazon S3 ?) dans la méthode save de l’object rendered_post_table.

Ensuite, afin de servir ces fichiers statiques, utilisez les capacités de votre framework web de servir des fichiers statiques (ils le proposent presque tous). Mieux encore, pourquoi ne pas servir ces fichiers avec votre serveur Nginx ?

Cette solution a les avantages de la précédente qu’elle combine avec ceux-ci :

  • Une seule connexion réseau est effectuée;
  • La base de données ne subit pas de requêtes;
  • Vous ne devez pas écrire de code applicatif pour “servir” les fichiers générés;
  • Le retour des données aux utilisateurs est encore plus rapide et plus léger.

Lorsque j’ai parlé de cette idée avec un senior developper, il m’a toutefois mentionné les désavantages suivants :

  • De la complexité accidentelle est introduite (Keep It Simple And Stupid);
  • On perd un peu des attributs de testabilité;

Si je comprends chacun de ces possibles désavantages, je ne suis pas forcément d’accord. En effet :

  • Le fait que stocker les données dans une base de données et de les render pour les fournir n’est selon moi pas “plus simple” que de servir statiquement des fichiers statiques; La solution est plus simple parce qu’elle est plus simple en ça qu’elle est plus commune. Autrement, je la trouve plus complexe, plus de code doit êtr écrit, testé, vérifié et peut finalement créer des instabilités;
  • Les attributs de testabilité ne sont selon moi pas perdus car il est possible de tester que le fichier généré a bien été écrit et contient les données appropriées. Et dans le cas d’un upload vers Amazon S3, la requête vers ce prestataire peut elle aussi être testée.

Ainsi, savoir si la seconde solution vaut selon vous la peine d’être mise en place est une question de choix. Les avantages et les inconvénients de sa mise en place doivent être pesé pour votre situation particulière. Elle reste cependant et selon moi la plus logique et la plus économiquement et écologiquement avisée.
